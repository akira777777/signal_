# Signal Group Sender

Небольшой CLI для контролируемой отправки одного сообщения в несколько заранее
разрешённых групп Signal.

Важно: у Signal нет официального публичного Bot API. Проект использует неофициальный
`signal-cli` через контейнер `signal-cli-rest-api`. Это подходит для внутренних
уведомлений и групп, участники которых согласились получать такие сообщения. Не
используйте инструмент для спама и не публикуйте API в интернет.

## Что реализовано

- локальная графическая панель на `http://127.0.0.1:8787`;
- вход по паролю и подписанная HttpOnly-сессия на 8 часов;
- выбор групп, редактор сообщения, dry-run и подтверждение в браузере;
- таймер отправки на 1, 5, 15, 30 или 60 минут с обратным отсчётом и отменой;
- список разрешённых групп хранится локально в `groups.json`;
- dry-run используется по умолчанию;
- реальная отправка требует `--execute` и точного `--confirm-count`;
- число групп и длина сообщения ограничены;
- между группами выдерживается интервал, действуют часовая/суточная квоты;
- повтор того же сообщения в ту же группу блокируется в течение заданного окна;
- параллельные запуски блокируются;
- текст сообщения и номер телефона не записываются в журнал;
- повторные рассылки выполняются серверным фоновым worker'ом и переживают
  закрытие вкладки браузера;
- POST-запрос отправки не повторяется автоматически при timeout, потому что
  результат доставки в такой ситуации неизвестен.

## Требования

- Python 3.11 или новее;
- Docker Desktop либо Docker Engine с Compose;
- работающий Signal на телефоне;
- свободный слот связанного устройства Signal.


## Установка

```powershell
Copy-Item .env.example .env
Copy-Item groups.example.json groups.json
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -e ".[dev]"
```

Укажите свой номер в `.env` в формате E.164.

Задайте пароль панели длиной не менее 4 символов:

```dotenv
SIGNAL_WEB_PASSWORD=1111
```

`1111` удобно для локального запуска, но это слабый пароль. Не публикуйте панель
в локальную сеть или интернет с таким паролем.

Запустите bridge без публикации API на host:

```powershell
docker compose up -d signal-api dashboard
```

Контейнер закреплён на `bbernhard/signal-cli-rest-api:0.100-rootless`. В обычном
режиме его порт доступен только другим контейнерам Compose, поэтому локальные
процессы не могут обойти allowlist и лимиты CLI прямым вызовом API.

## Графическая панель

После запуска откройте:

```text
http://127.0.0.1:8787
```

Войдите с паролем `SIGNAL_WEB_PASSWORD` из `.env`.

Основной сценарий:

1. В боковой панели выберите активный Signal-аккаунт.
2. Чтобы добавить другой аккаунт, нажмите «Подключить другой аккаунт» и
   отсканируйте QR-код в Signal на телефоне.
3. Выберите нужные группы или нажмите «Выбрать все».
4. Введите сообщение и при необходимости добавьте изображения.
5. Укажите количество циклов отправки от 1 до 20.
6. Для нескольких циклов задайте интервал между ними.
7. Нажмите «Проверить отправку», проверьте получателей, изображения, число
   циклов и код.
8. Нажмите «Отправить».

Поддерживаются PNG, JPEG, WebP и GIF. Максимальный размер одного изображения —
8 МБ, суммарный размер — 20 МБ. MIME-тип, base64 и сигнатура файла повторно
проверяются сервером. Для фоновых повторов текст активной кампании, выбранные
получатели и вложения сохраняются в локальном файле `*-campaign.json` рядом с
state-файлом и включаются в код подтверждения рассылки. После завершения или
отмены кампания остаётся в этом файле как последний статус запуска без записи в
журнал отправлений.

Панель получает группы непосредственно из выбранного связанного аккаунта.
`groups.json` и его SHA-256 по-прежнему обязательны для CLI-команд, но не
ограничивают список групп в графической панели. Для панели сохраняются лимит
групп, почасовые/суточные квоты, cooldown и явное подтверждение.

Интервалы выполняются сервером: после запуска кампании вкладку можно закрыть,
а следующий цикл всё равно будет выполнен контейнером `dashboard`. Для примера,
50 циклов с интервалом 30 минут будут последовательно выполняться фоновым
worker'ом, пока работает контейнер и компьютер не находится в сне. При
перезапуске панели незавершённая кампания подхватывается из локального
`*-campaign.json`. Кнопка «Остановить повторы» отменяет следующие циклы, но не
прерывает уже начатый запрос. После перезапуска панели активным снова становится
`SIGNAL_NUMBER` из `.env`.

Остановить панель и bridge:

```powershell
docker compose down
```

## Привязка к Signal

Проще всего подключить новый аккаунт прямо в графической панели кнопкой
«Подключить другой аккаунт».

Для первичной привязки получите QR-код через одноразовый контейнер в закрытой
Docker-сети:

```powershell
docker compose up -d signal-api
docker compose --profile link run --rm linker -fsS `
  "http://signal-api:8080/v1/qrcodelink?device_name=signal-group-sender" `
  -o /output/signal-link.png
```

Откройте `outputs\signal-link.png`.

На телефоне откройте Signal → Настройки → Связанные устройства → Добавить и
отсканируйте QR-код. QR-код короткоживущий; при необходимости обновите страницу.

## Настройка групп

Получите группы, видимые связанному аккаунту, через изолированный CLI-контейнер:

```powershell
docker compose --profile cli run --rm broadcaster groups
```

Команда выводит имя, доступность и идентификатор вида `group.…`. Перенесите
только нужные идентификаторы в `groups.json` и назначьте им короткие aliases:

```json
{
  "operations": {
    "id": "group.ACTUAL_ID_FROM_COMMAND",
    "description": "Операционные уведомления"
  }
}
```

Файл `groups.json` — allowlist. Команда отправки не принимает произвольный номер
или идентификатор группы из аргументов.

Закрепите allowlist в `.env`:

```powershell
(Get-FileHash .\groups.json -Algorithm SHA256).Hash.ToLower()
```

Скопируйте результат в `SIGNAL_ALLOWLIST_SHA256`. Реальная отправка блокируется,
если `groups.json` был изменён.

## Отправка

Сначала выполните dry-run:

```powershell
Get-Content .\message.txt -Raw | docker compose --profile cli run --rm -T `
  broadcaster send --group operations --stdin
```

CLI покажет aliases и длину сообщения, но не сам текст.
Также dry-run выводит `confirm_token`, привязанный к точным group ID и тексту.

Реальная отправка в одну выбранную группу:

```powershell
Get-Content .\message.txt -Raw | docker compose --profile cli run --rm -T `
  broadcaster send --group operations --stdin --execute --confirm-count 1 `
  --confirm-token TOKEN_FROM_DRY_RUN
```

Несколько групп:

```powershell
Get-Content .\message.txt -Raw | docker compose --profile cli run --rm -T `
  broadcaster send --group operations --group team --stdin `
  --execute --confirm-count 2 --confirm-token TOKEN_FROM_DRY_RUN
```

Все группы из allowlist:

```powershell
Get-Content .\message.txt -Raw | docker compose --profile cli run --rm -T `
  broadcaster send --all-allowed --stdin --execute --confirm-count 2 `
  --confirm-token TOKEN_FROM_DRY_RUN
```

Число в `--confirm-count` должно совпадать с фактическим числом адресатов. После
частичной рассылки повтор той же команды автоматически пропускает группы со
статусом `already_sent` и продолжает непопытанные. Если предыдущий результат
`delivery_unknown`, сначала проверьте Signal вручную; для осознанного повтора
только такой группы добавьте `--retry-unknown`.

Для автоматического запуска используйте `--message-file` или `--stdin`, а сам
запуск оформите через Windows Task Scheduler, cron либо ваш CI. Опции передачи
текста прямо в аргументе нет: аргументы процесса могут быть видны другим
локальным пользователям.

## Эксплуатационные ограничения

- Dashboard публикуется только на `127.0.0.1:8787`; не меняйте binding на
  `0.0.0.0`, если не добавили TLS и сетевую аутентификацию.
- Пароль панели хранится локально в `.env`. Не добавляйте этот файл в Git и
  замените примерный пароль перед использованием.
- Web-сессия действует 8 часов, использует HttpOnly/SameSite cookie и
  подписывается локальным секретом из Docker volume.
- API не имеет встроенной аутентификации. Основной Compose-файл не публикует его
  на host. И broadcaster, и QR-linker обращаются к нему только через внутреннюю
  Docker-сеть.
- Каталог/volume `signal-cli` содержит ключи связанного устройства. Не копируйте
  его в репозиторий и ограничьте доступ к Docker-хосту.
- При timeout, HTTP-ошибке или неожиданном ответе отправка помечается как
  `delivery_unknown` и не повторяется
  автоматически. Проверьте группу вручную перед повторным запуском.
- Signal может отвязать устройство после длительной неактивности. Проверяйте
  список связанных устройств на телефоне.
- Ledger подписан HMAC и его удаление после инициализации блокирует отправку.
  Однако локальный администратор всё равно может откатить весь Docker volume к
  старому согласованному snapshot; защищайте backup и доступ к Docker-хосту.

## Используемые интерфейсы

- `GET /v1/groups/{number}` — получить список групп;
- `GET /v1/accounts` — получить связанные аккаунты;
- `POST /v2/send` — отправить сообщение, указав `recipients: ["group.…"]`;
  изображения передаются массивом `base64_attachments`;
- `GET /v1/qrcodelink` — связать контейнер с мобильным аккаунтом.
